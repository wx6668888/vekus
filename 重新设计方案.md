# Vekus 智能报价平台 · 重新设计方案

> 版本:v1.0(2026-05-10)
> 范围:视觉语言、信息架构、前端代码骨架、报价工作台详细设计、组件库、实施路线
> 适用:替代当前 `index.html` 单页原型,作为正式开发前的基线

---

## 0. 一句话定位

**Vekus 是给钣金老板和业务员用的"图纸即报价"工作台,客户用一个链接就能看到最终价格。**

围绕这一句,所有页面、组件、交互必须服务三件事:**更快出价、更准识别、更顺成交**。

---

## 1. 当前问题诊断

| 类别 | 现状 | 问题 |
|---|---|---|
| 视觉 | `index.html` 是深色沉浸+毛玻璃 orb 风,但 `设计规范.md` 描述的是"浅色 SaaS+深色 Hero" | **风格自相矛盾**,无法同时作为"工厂工具"和"客户分享页" |
| 架构 | 仓库根目录只有一个 `index.html`,文档却描述 9+ 个页面 | **代码与文档严重脱节**,新成员上手困难 |
| 主链路 | 报价工作台用模拟数据 + 卡片骨架展示,缺真实计算与状态机 | **核心业务未跑通**,看起来像 demo 不像产品 |
| 角色 | 文档区分老板/业务员/客户/管理员,代码无任何角色分流 | **多角色协同的产品价值无法体现** |
| 移动端 | 桌面端为主,手机端是收缩单栏 | 业务员实际是在工厂里用手机拍图开单,移动端应该是**一等公民** |
| 数据可信度 | 报价数字是大字号,但**字体不分等级**,17.99 和 17,990 一眼分不清 | 报价类产品**数字必须用等宽数字字体**,这是底线 |
| 交易模块 | 既要做"库存交易撮合"又要做"报价工具",文档自己也犹豫 | **范围不清**,建议本期降级为"次级模块",先把报价做到极致 |

## 2. 设计 DNA:精工蓝图(Precision Blueprint)

### 2.1 灵感来源

钣金行业的两个核心意象:
- **工程图纸**:网格、尺寸线、剖面、公差,强调精确
- **现代精工车间**:冷静、克制、高对比度数据,强调可信

这两者合起来 → 一个**"看起来就像在读图纸,但操作起来像现代 SaaS"**的视觉系统。

### 2.2 设计原则(三条铁律)

1. **数字优先于装饰**
   报价金额、单价、利润率、识图参数 —— 这些数字是产品本体。所有动效、配色、布局都不能抢这些数字的注意力。

2. **三秒读懂当前状态**
   任何页面在 3 秒内必须告诉用户:这是哪个客户、当前金额是多少、下一步该做什么。看板、报价、历史、客户详情 —— 全部遵循这一条。

3. **手机端不是"缩小版"**
   业务员在工厂里单手操作,关键动作(拍照上传、调参、发送)必须能在拇指热区单手完成。手机端布局**单独设计**,而不是把桌面端折叠。

### 2.3 风格关键词

精确 · 克制 · 工业 · 可信 · 高密度可读 · 蓝图感

### 3.3 间距与圆角

间距用 4px 基准:`4 / 8 / 12 / 16 / 20 / 24 / 32 / 48 / 64`。

圆角层级:
- 微元素(徽章、Tag):`6px`
- 表单控件、按钮:`10px`(比常见 14px 略小,显得"工具感"而非"消费品感")
- 卡片:`16px`
- 大区块/Hero:`20px`
- 抽屉/Modal:`24px`


### 3.5 蓝图底纹(品牌资产)

深蓝 Hero 区(`--surface-blueprint`)默认叠加一层**工程网格 + 微噪点**:

```css
background:
  linear-gradient(180deg, #0B1C3A 0%, #102B5C 100%),
  url("data:image/svg+xml;utf8,<svg xmlns='http://www.w3.org/2000/svg' .../>"); /* 4px × 4px 网格 */
```

这一笔是整个品牌的视觉记忆点。客户收到分享链接打开报价单时,第一眼看到的就是这片"图纸蓝"。

## 4. 信息架构

### 4.1 角色 → 入口 映射

| 角色 | 默认落地页 | 主导航(从左到右) | 核心 KPI |
|---|---|---|---|
| 老板 | `/dashboard`(经营看板) | 看板 / 报价 / 客户 / 参数 / 后台 | 转化率、利润率 |
| 业务员 | `/quote`(直接进报价) | 报价 / 历史 / 客户 / 我的 | 当日报价数、待跟进 |
| 客户 | `/share/:id`(分享页) | 报价详情 / 联系业务员 | 报价金额一目了然 |
| 管理员 | `/admin`(后台首页) | 用户 / 参数 / 日志 / 系统 | 系统健康度 |

**关键决策**:登录后**按角色直达工作场景**,而不是统一进首页让用户再点一次。

### 4.3 导航模型

**桌面端**:左侧固定侧边栏(240px),只有 5-6 个一级入口,深度模块在右侧主区分级展示。

**手机端**:底部 Tab Bar(4 个 + 1 个中央凸起的"+"主动作)
```
[首页]  [历史]   [+]   [客户]  [我的]
                ↑
            浮起的"新建报价"按钮(48×48 圆形,品牌橙)
```

中央 + 按钮直接进入"拍照/选图 → 识别 → 报价"全屏流程,这是业务员最高频动作,值得占据视觉中心。

## 5. 报价工作台详细设计(本期重点)

### 5.1 设计目标

- 业务员从"开始建单"到"生成可分享报价"≤ **90 秒**
- 任何参数变更后金额刷新延迟 ≤ **150ms**
- 移动端单手可完成全流程

### 5.2 三种形态

| 形态 | 何时出现 | 布局 |
|---|---|---|
| **桌面工作台** | ≥ 1280px | 左侧表单(60%) + 右侧粘性结果面板(40%) |
| **平板工作台** | 768-1279px | 上下分区:上方表单,下方折叠的结果浮窗 |
| **手机工作台** | < 768px | 步骤化分屏(基础 → 上传 → 修正 → 确认),底部固定金额 + 主按钮 |

#### 关键交互

- **任意输入即重算**(debounce 80ms),右侧金额无刷闪,数字滚动到位
- **AI 识别项右侧带 `✎`**,点击转编辑模式,人工修正后该字段加 `已校正` 角标(用 `--accent` 橙色),便于复盘
- **图纸上传支持拖拽 + 粘贴**(`Ctrl+V`),工厂场景里业务员经常微信收图后直接粘贴
- **顶部"保存草稿/发送"按钮粘性悬浮**,滚动不消失
- **结果面板粘性**,不论左侧表单滚到哪都能看到金额

### 5.5 报价计算引擎(前端契约)

```ts
// 输入
type QuoteInput = {
  basics: { material: string; thickness: number; quantity: number; surface: string };
  recognized: {  // AI 给的初值
    expandLength: number; expandWidth: number;
    cutLength: number; bendCount: number;
    paintArea: number; weldPoints: number; weldLength: number;
    holes: { plain: number; threaded: number; counterbored: number };
  };
  manualOverrides: Partial<typeof recognized>;  // 业务员修正的字段
  coefficients: { tax: number; discount: number; packaging: number; profitRate: number };
};

// 输出
type QuoteResult = {
  breakdown: { material; cutting; bending; welding; surface; admin; profit }; // ¥
  totalCost: number;
  totalPrice: number;
  unitPrice: number;
  profitMargin: number;  // %
  warnings: string[];    // 例:"折弯数 > 12,建议人工复核"
};
```

计算函数纯函数化、无副作用,放在 `src/services/quoteCalculator.ts`,**前端先有兜底实现**,后端接入后切换到 `/api/quote/calculate` 调用同样的契约。

### 5.6 状态机

```
[空白] → [填基础] → [上传中] → [识别中] → [可编辑] ⇄ [修正中]
                                              ↓
                                          [可发送] → [已发送] → [客户已查看] → [已成交 / 已失单]
                                              ↓
                                          [保存草稿]
```

每次状态切换写一条 `activity_log`,客户查看分享页时回写 `customer_viewed_at`,业务员能看到"客户已读"的微信式回执。

## 7. 前端代码骨架建议

### 7.1 推荐技术栈

| 决策 | 推荐 | 理由 |
|---|---|---|
| 框架 | **Vite + 原生 JS Module**(轻)或 **Vite + Vue 3**(中型)或 **Next.js 14**(完整) | 看团队 |
| 样式 | **CSS 变量 + PostCSS**,不用 Tailwind | 设计系统强,Tailwind 会冲淡品牌资产 |
| 状态 | URL + localStorage + 极简 store(Pinia / Zustand) | 报价业务天然适合 URL 驱动 |
| 数据 | fetch wrapper,所有接口收口在 `src/api/` | 替换后端时只改一处 |
| 图标 | Lucide(线性,工程感) | 不用 Material(太消费品) |
| 图表 | ECharts 轻量版 / @observablehq/plot | 仪表盘够用 |

我的**强推荐**:**Vite + Vue 3 + Pinia + 自研组件库**。原因:
- Vue 模板对中文团队上手成本最低
- 原型阶段已有 `src/` 模块化结构,迁移代价小
- 报价页大量"输入即重算"场景,Vue 的响应式天然适配

如果坚持原生 JS(像现在),可以接受,但需要把 `src/` 重构为更清晰的分层。

## 8. 关键页面设计要点(快速过)

### 8.1 经营看板 `/dashboard`(老板视角)

- **顶部 4 个 KPI 卡**:今日报价数、待跟进客户、本周成交额、平均利润率;每个右下角带"7 日趋势 spark line"
- **中部双栏**:左 = 报价金额折线图(可切日/周/月);右 = 业务员排行榜(头像 + 报价数 + 成交率)
- **底部任务队列**:待审核报价 / 待确认客户 / 异常识别需复核

### 8.2 历史报价 `/history`

- **顶部 sticky 工具栏**:搜索框 + 状态 Tab + 业务员筛选 + 日期范围
- **DataTable**:报价单号(等宽)| 客户 | 材料 | 数量 | 金额(等宽,右对齐) | 状态 Badge | 业务员 | 时间 | 操作
- **每行 hover 出"复用 / 详情 / 分享"按钮**
- **空状态**:大插图 + "新建第一单"主按钮

### 8.3 客户管理 `/customers`

- 左侧客户列表(支持字母索引),右侧详情**抽屉式**展开
- 详情含:基础资料 | 标签 | 历史报价时间线 | 沟通记录 | 业务员
- **关键差异**:不要做"客户表格",而是做"客户档案"—— 让业务员 1 秒回忆起客户是谁

### 8.4 客户分享页 `/share/:id`(无登录)

这是**对外的脸面**,优先级仅次于报价工作台。

- 顶部:Vekus 品牌条(轻量,不抢戏)
- 主体:**蓝图 Hero**,巨大的 ¥金额,客户名 + 报价单号 + 业务员名片
- 中部:配置摘要(材料/厚度/数量/工艺) + 简化后的成本拆解(可折叠,默认折叠 —— 客户多数只关心总价)
- 底部:CTA "确认下单 / 联系业务员 / 索要发票"
- 必须**移动端优先**:多数客户在微信里点开

### 8.5 登录页 `/login`

去掉花哨的 hero,**直接居中表单卡片**,左侧大面积蓝图底纹 + Vekus logo。
- Tab:手机验证码 / 微信扫码
- 测试账号在浅灰色 footer,不显眼

### 8.6 参数设置 `/settings`

左侧分类树(材料/工艺/系数/规则),右侧表格 + 行内编辑。**新增参数用顶部 inline 表单**,不开 Modal,降低操作成本。

## 11. 风险与取舍

| 风险 | 取舍建议 |
|---|---|
| 框架升级到 Vue 3,旧 `index.html` 完全推翻 | 推翻是必要的 —— 现状是 demo,继续叠加成本更高 |
| 蓝图深色 Hero 在低端屏幕上对比度问题 | 主操作色 `--brand` 与白色文本在 `#0B1C3A` 上对比度 12:1,符合 WCAG AAA |
| 等宽字体对中文用户略陌生 | 仅用于数字/编号,中文仍走 Inter+苹方,不影响阅读 |
| 交易市场降级,可能与既有期望冲突 | 文档中保留入口,但移到次级位置;一期不投入设计资源,二期再做 |
| 移动端单独设计成本高 | 这是必要投入 —— 业务员实际就是手机用户,对齐桌面 UX 是性价比最低的选择 |
